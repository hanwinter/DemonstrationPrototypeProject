请修复家长端移动端“复筛通知”页面的路由来源与返回路径问题，并顺带统一检查多入口页面的返回逻辑。

当前问题：
从“报告页面”点击“查看通知”进入“复筛通知”页面后，点击返回时没有回到报告页面，而是进入了“首页快捷入口进入复筛通知”后的返回路径，导致返回目标错误。

原因判断：
“复筛通知”页面是多入口页面，目前返回逻辑可能被写死为首页，或只记录了首页快捷入口来源，没有正确识别从报告页面进入的场景。

一、复筛通知页面入口来源规则

“复筛通知”至少有两个入口：

1. 首页快捷入口 → 复筛通知
2. 报告页面“查看通知” → 复筛通知

请根据入口来源处理返回路径：

1. 从首页快捷入口进入复筛通知：
   - 返回时应回到首页
   - 底部 Tab 应保持“首页”选中状态

2. 从报告页面点击“查看通知”进入复筛通知：
   - 返回时应回到报告页面
   - 底部 Tab 应保持“报告”选中状态
   - 不要跳转到首页
   - 不要走首页快捷入口的返回路径

二、路由实现要求

1. 不要将复筛通知页面的返回按钮写死为首页。
2. 建议通过 query / state / meta / router history 记录来源页面。
3. 推荐方式之一：
   - 首页快捷入口跳转复筛通知时携带：
     from=home
   - 报告页面点击查看通知跳转复筛通知时携带：
     from=report
4. 示例：
   - /rescreen-notice?from=home
   - /rescreen-notice?from=report
5. 如果涉及学生 ID、报告 ID、通知 ID，也请保留原有参数，不要破坏业务数据传递。
6. 如果当前项目使用 router.back() 更合适，也可以优先使用 router.back()，但必须保证上述两个入口返回结果正确。

三、复筛通知详情页返回逻辑

如果“复筛通知”页面下还有通知详情页，请同时处理：

1. 复筛通知详情页返回：
   - 应先回到复筛通知列表页。
2. 复筛通知列表页再根据来源返回：
   - from=home → 首页
   - from=report → 报告页面
3. 不要从详情页直接跳回首页，除非它本来就是首页进入并且当前交互明确要求直接返回首页。

四、报告页面进入复筛通知的特殊要求

从报告页面进入复筛通知后，返回时应保持报告页面上下文：

1. 回到报告页面。
2. 底部 Tab 保持“报告”选中。
3. 尽量保留报告页面原有滚动位置或当前学生/报告状态。
4. 不要重置为首页状态。

五、统一排查多入口页面

本项目中已经出现多个多入口页面返回路径错误，请顺带检查并修复类似问题：

1. 就诊人管理
   - 首页进入 → 返回首页
   - 我的进入 → 返回我的

2. 复筛通知
   - 首页进入 → 返回首页
   - 报告进入 → 返回报告

3. 问卷量表
   - 首页快捷入口进入 → 返回首页
   - 底部 Tab 问卷进入 → 返回问卷 Tab 页面

4. 报告详情 / 报告相关页面
   - 首页最近检查进入 → 返回首页
   - 报告 Tab 进入 → 返回报告 Tab 页面

六、实现原则

1. 所有多入口页面必须保留入口来源。
2. 返回按钮不能统一写死到首页。
3. 返回目标应符合用户进入路径。
4. 底部 Tab 选中状态要和返回页面保持一致。
5. 修复时不要影响现有页面 UI 样式。
6. 不要改变复筛通知的业务内容，只修复路由和返回逻辑。
7. 如果项目已有统一导航方法，请优先复用统一方法，避免每个页面单独写一套返回逻辑。

场景一：
首页 → 复筛通知 → 返回

场景四：

结果：先回到就诊人管理，再回到我的页面。
